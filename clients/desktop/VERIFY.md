# v0id 浏览器 · 操作与验证书

在你自己的 Mac 上**手动验证** GUI：本地起一个 demo `.v0id` 隐藏服务网络，再让 Electron 浏览器经它的
SOCKS5 访问那个 `.v0id`，亲眼看到页面渲染出来。

> 这是「**本地 demo 网络 + 浏览器**」的离线验证：**全程不碰公网链**、不挖矿、纯进程内内存中继。
> 它验证的正是 GUI 将走的那条路：webview → SOCKS5（远程 DNS）→ rendezvous → 隐藏服务。
> （公网真链验证见文末「进阶」，留作后续。）

---

## 前置

- **Node**：已有（仓库工具链用 `corepack pnpm exec tsx` 跑 TS）。
- **worktree 路径**：本仓库这个 worktree 的根目录，下文记作 `<根>`，即包含 `clients/desktop/` 的那一层。
  所有「终端 1」命令都从 `<根>` 跑。
- **联网**：仅「终端 2」首次 `pnpm install` 需要联网下载 Electron 二进制（~100MB，走 GitHub CDN）；
  demo 网络与访问过程**全本机回环，不需要公网**。

## 步骤（两个终端）

### 终端 1 —— 起 demo 网络（保持运行）

在 `<根>` 下：

```bash
corepack pnpm exec tsx clients/desktop/scripts/demo-network.mjs
```

它会起 6 个进程内中继 + 一个隐藏服务（背后桥到本机 HTTP）+ 一个 SOCKS5（默认 `:9050`），然后打印横幅：

```
────────────────────────────────────────────────────────────────
  v0id demo 网络就绪，保持本终端运行
────────────────────────────────────────────────────────────────
  要在浏览器里访问的地址：  http://xxxxxxxx.v0id/
  .v0id 地址：              xxxxxxxx.v0id
  SOCKS5 端口：             9050
────────────────────────────────────────────────────────────────
```

**记下那一行 `xxxxxxxx.v0id` 地址**（每次启动随机生成，不一样）。**这个终端保持运行别关。**

### 终端 2 —— 起浏览器，指到 demo 的 SOCKS

```bash
cd clients/desktop
pnpm install --ignore-workspace        # 首次：装 React/Vite/electron（务必带 --ignore-workspace，见排错）
pnpm build                             # 构建 React 渲染层 → src/renderer/dist/（首次或改过 .jsx 后必跑）
V0ID_SOCKS_EXTERNAL=9050 pnpm start     # 让浏览器用终端 1 已起的 SOCKS:9050，而不自起守护
```

浏览器窗口打开后：左侧栏在「浏览器 / 浏览客户端 / 中继 / 托管站点 / 链·挖矿 / 钱包」之间切换，
默认停在**「浏览器」板块**。底部状态行应显示 **「外部 SOCKS :9050（demo 网络）」**（绿点 = 就绪、可导航）。
新标签起始页会提示「外部 SOCKS 验证模式已启用」。

在地址栏**粘贴终端 1 那个 `.v0id` 地址** → 回车。

> 想用热更新改 UI：`pnpm dev`（终端 A）+ `V0ID_SOCKS_EXTERNAL=9050 V0ID_RENDERER_DEV_URL=http://localhost:5173 pnpm start`（终端 B）。

## 验证清单（看到上面那页 = 三条同时成立）

1. **页面正确渲染。** `.v0id` 不是真实 TLD，本机 DNS 解析不了它。能渲染出来，就证明 webview 经
   `socks5://` 代理把主机名**原样交给了 SOCKS 做远程 DNS**（等价 `curl --socks5-hostname`），
   而不是本机解析失败——这是浏览器接线正确的核心判据。

2. **终端 1 出现该次访问的活动日志。** 收到访问的瞬间，**终端 1** 会打印一行：

   ```
   HH:MM:SS  [hs] 收到请求 GET /  ← 一次 rendezvous 会合已送达此本机落地
   ```

   这个本机 HTTP 落地**只有经 rendezvous 桥接才连得到**（没有任何对外端口），所以这行 = 请求**真的走了会合**、
   而非直连。
   （说明：底层中继/会合模块本身是静默的、不打日志；这条「收到请求」是 demo 在它**自己掌控**的落地服务上加的
   可观察探针——它落到这里，就只可能是经会合到达的。）

3. **匿名性推理。** 浏览器全程只有那个 `.v0id` 地址、经 rendezvous 连上，**自始至终不知道隐藏服务的 IP**；
   反过来隐藏服务也只看到会合点、不知道你的 IP。这正是 `.v0id` 的双向匿名属性。

## React 外壳 + 浏览器板块 GUI 清单（在 Mac 上亲手点一遍）

渲染层已从单 webview 改成 React 多板块外壳。除了上面那页能渲染出来，还请逐项确认：

1. **左侧栏 6 个板块都可切换**：浏览器 / 浏览客户端 / 中继 / 托管站点 / 链·挖矿 / 钱包。
   后 5 个是占位面板——显示角色说明 + 「下一阶段 (E)」提示；自起守护（不带 `V0ID_SOCKS_EXTERNAL`）时
   还会显示只读链状态（链高/对等）。外部 SOCKS 模式下链状态显示「不可用」（正常，demo 网络无链 API）。
   鼠标悬停「对等」二字应弹出提示：这是 P2P 广播连接数，跟「中继」板块的可达中继数是两套独立机制
   （开中继不会让这个数变大）——两者数字不一致是正常的，不是 bug。
2. **多标签**：点标签条右侧 `+` 新建标签；每个标签独立加载、可单独关闭（`✕`）；切回旧标签其内容仍在
   （非活动标签只是隐藏、未销毁）。
3. **前进/后退/刷新**：在一个标签里先访问 demo 的 `.v0id`，再点站内链接（或手动导航到别处）后，
   「‹」可后退、「›」可前进，到边界时按钮置灰；加载中刷新键变「✕」（停止）。
4. **地址校验**：乱输（如 `hello`）→ 出现「地址无效」覆盖层；合法 `.v0id` 或 `http(s)://` → 正常加载。
5. **书签**：访问到一个页面后，点地址栏右侧 ☆ → 变实心 ★（已收藏）；新建一个标签，其起始页「书签」区
   应出现刚才那条，点它能打开；点书签卡角的 `✕` 可删除。**关掉应用再开**，书签仍在
   （存于 `userData/bookmarks.json`）。
   - **跨标签书签入口**：收藏后开一个新标签页（不回起始页），且**该标签已经加载了某个页面**（不是空白
     起始页——这是复现下面那个 bug 的关键），点工具栏地址栏右侧的书签按钮（☰），应弹出悬浮列表列出所有
     书签，**完整可见、不被下方网页内容遮住**（`<webview>` 是独立合成层，历史上无论 CSS z-index 多高都会
     画在浮层上面，导致列表下半截被盖住看不见；现在弹出期间会临时隐藏当前网页内容），点其中一条能直接
     跳转、无需先回到起始页。
   - **地址栏自动补全**：同样先导航到某个页面后，在地址栏输入几个字符（能匹配某条书签的 URL 或标题），
     应弹出下拉建议列表（书签项带 ★ 标记）、同样完整可见不被网页盖住；方向键可上下选中，回车跳转到选中项；
     输入没有任何匹配时不弹出下拉，回车按原字符串走正常导航逻辑。
   - **会话态 vs 持久态边界**：先访问几个不同地址攒一些「最近访问」，重启应用后确认书签还在，
     但地址栏联想里不再出现刚才那些非书签的「最近访问」项（只在本次会话内存活，不落盘）。
6. **起始页诚实**：新标签起始页只列你自己的书签 + 引导文案，**不含任何预置的假 `.v0id` 地址**。
7. **历史/会话默认不落盘**：`.v0id` 页面用**内存型** partition（`v0id`，无 `persist:` 前缀），
   cookie / 缓存 / localStorage / 已访问链接全随进程退出蒸发——磁盘上不留浏览痕迹。
   起始页底部「本次会话最近访问」也只在本次运行内有效、**关窗重开后清空**。唯一落盘的是你主动收藏的书签。
   （可旁证：访问后查 `~/Library/Application Support/v0id-browser/`，不应出现含 `.v0id` 主机的持久 Partitions 数据。）
8. **弹窗/越权被拦**：`.v0id` 页面里的 `window.open` / `target=_blank` 一律被拒（不会弹出未加固的新窗口）；
   地址栏也拒绝环回/私网地址（如 `127.0.0.1`、`localhost`、`192.168.*`），防止恶意页诱导去打本机守护 API。

## Phase E：角色板块 + 钱包（需自起守护，连真链）

> 上面的 demo 验证用 `V0ID_SOCKS_EXTERNAL=9050`（外部 SOCKS、**不起守护**）——那种模式下**没有本机节点 API**，
> 角色开关/质押/钱包会显示「外部 SOCKS 验证模式，无本机节点 API」。要验证 Phase E 的运行时控制，
> 必须让浏览器**自起守护进程**（不带 `V0ID_SOCKS_EXTERNAL`），它会连出厂种子或你给的 `V0ID_PEERS`。

### 起法（自起守护）

```bash
cd clients/desktop
pnpm install --ignore-workspace
pnpm build
pnpm start                              # 自起守护：连出厂种子（seeds.js）；数据/令牌在 userData/v0id/
# 或指定本地种子：V0ID_PEERS=ws://<种子>:6001 pnpm start
```

守护起来后（底部状态行变绿「SOCKS 已就绪」），左侧切到各角色/钱包板块亲手点：

1. **浏览客户端**：显示 SOCKS 端口 + 状态（运行中），链高/对等。无开关（这是始终开的基座）。
2. **链 · 挖矿**：
   - 看到链高/区块/对等/内存池/余额/同步态（每几秒刷新）。
   - 填「出块间隔(ms)」（默认 8000）→ 点「开始挖矿」→ 按钮变「● 挖矿中」，链高开始上涨、余额随出块增长。
   - 点「停止」→ 回「○ 未挖矿」。
3. **中继**：
   - 点开关「上线」→ 变「● 中继运行中」，显示 cell 端口、活动电路、描述符上链状态（余额≥2 时自动发布）。再点「下线」回 ○。
   - **三段式状态 + 可达性自测**：开关下方应出现三行独立指示——「本地服务已启动」「描述符已上链发布」
     「可达性自测」。默认广播地址是回环 `127.0.0.1` 时，「已上链发布」应保持灰（待发布）——这是刻意设计
     （回环地址对外不可达，发布只会污染全网中继目录）。**同一台机器探测自己的回环地址必然连得通**，
     所以这种情况下点「测试是否可达」应直接报错「广播地址是回环/私网……自检没有意义」，而不是谎报绿灯——
     不然会误导你以为真的上线了。用 `--relay-advertise` 指到一个真实公网地址/域名重跑一遍，
     「已上链发布」应变绿，这时点自检才会真正探测那个公网地址、成功则变绿。下线中继后再点自检按钮应报错
     （中继未上线，无法自检）。
   - **广播地址 GUI 配置**（不用再手动传 `--relay-advertise` 重开 CLI）：「广播地址」区块显示「当前生效：
     127.0.0.1:<端口>」；填一个公网地址/域名（或域名+自定义端口）后点「保存并重启守护进程」，状态提示
     「重启中…」→「已重启守护进程，新的广播地址已生效」，几秒后「当前生效」应变成刚填的值，且如果余额
     ≥2，「描述符已上链发布」应转绿。重启期间当前浏览标签页/托管服务会短暂中断（预期行为，非 bug）。
     清空输入框再保存+重启应能回退到默认 127.0.0.1。
   - **重启后中继角色应自动恢复**：先点「上线」把中继开着，再改广播地址触发重启——重启完成后状态应显示
     「守护进程已重启，正在恢复中继上线状态……」最终变绿「已重启守护进程，新的广播地址已生效，中继已恢复
     上线」，中继开关应保持「● 中继运行中」，而不是重启完却悄悄掉线、界面却报「成功」。
   - **非法端口/外部 SOCKS 模式下的边界情况**：广播端口填 `0` 或 `70000` 之类非法值点保存，应直接报错
     「端口须为 1–65535 的整数」，不落盘也不触发重启。`V0ID_SOCKS_EXTERNAL` demo 验证模式下点「保存并
     重启守护进程」应报错「外部 SOCKS 验证模式下没有内部守护进程，无法重启」，而不是真的在 demo 的
     SOCKS 端口上又起一个内部守护进程打架。
   - **中继选路不再误伤钉住的入口守卫**（无头脚本可测，不需要真链）：在 `packages/node/src/relay/guards.ts`
     的 `GuardManager.currentGuard` 上写个小脚本——钉住 3 个守卫后，只传一个「排除了当前主守卫」的**已过滤
     可达集**（模拟一次瞬时探测失败）而不是全量目录，确认主守卫仍然待在 `guards.json` 里（用
     `allGuards()` 查）没有被删除；只有真的从**全量目录**里消失（模拟中继下线）才应该被剔除。
   - **全网中继统计**：状态卡格里的「全网中继」应显示「注册 N · 可达 M」两个数字，而不是单一数字；
     M ≤ N（N=链上曾登记过的全部地址，M=刚探测到确实连得上的数量）。
   - **质押门禁（诚实性核心）**：当前公网链高若 **< 16000**，质押区显示
     **「质押共识将于 height 16,000 激活（当前链高 …）」**，且**没有**角色选择器与质押按钮（被门禁替代）。
     这就是 `STAKING_ACTIVATION_HEIGHT` 的诚实呈现——激活前不让你把押金锁进托管却不生效。
   - 链高 ≥ 16000 后：出现角色下拉（guard 12 / hsdir 8 / middle 4）+「质押」按钮；选一个角色点质押 →
     提示「质押已提交（txid…）」；下方「本节点质押池」列出本金/锁定高度，锁定期内「赎回」按钮置灰为「锁定中」，
     过锁定期后可赎回。
   - 始终能看到**「引导期暂不发放奖励（见 INCENTIVE-PROTOCOL）」**与底部常驻的
     **「小网络=弱匿名 + v1 激励中心化」**提醒。
4. **托管站点**：
   - 先在另一个终端起个本机服务（如 `python3 -m http.server 8080`）。
   - 板块里目标默认 `127.0.0.1:8080` → 点「发布隐藏服务」。
   - 若链上中继 **< 3**：友好提示「链上中继不足 3 个，暂无法托管…」（对应后端 409）。
   - 若 ≥ 3：显示你的 `.v0id` 地址 + 「复制」按钮（可粘到「浏览器」板块访问验证），「停止托管」可下线。
   - **零后端静态托管**：建一个含 `index.html` 的临时文件夹，切到「本地文件夹（零后端）」模式，点
     「选择文件夹…」选中它，点「发布隐藏服务」——不需要自己起任何后端进程。发布成功后用「浏览器」
     标签页访问打印出的 `.v0id` 地址，应能看到该 `index.html` 的内容。服务卡片上应显示
     `📁 <文件夹路径>` 而非 `host:port`。停止后立即用另一个文件夹重新发布，确认能正常起来（端口未泄漏）。
     再试一次选一个**没有** `index.html` 的文件夹，应在选择后立刻出现「该文件夹没有 index.html……」提示。
   - **静态托管安全（无头脚本即可测，不需要 GUI）**：
     - 路径遍历/符号链接逃逸：在发布的文件夹里 `ln -s /etc/passwd escape.txt`，访问
       `<地址>/escape.txt` 应该 404（而不是把 `/etc/passwd` 的内容读出来给访问者）。
     - 两个不同文件夹先后发布分别拿到**不同**的 `.v0id` 地址（不会因路径里标点差异而被误判成同一个服务、
       把第二次发布悄悄指回第一个文件夹）。
   - **关窗即下线（确认文案与实际行为一致）**：本 App 关窗即退出并杀掉守护进程（`window-all-closed` →
     `killDaemon()` + `app.quit()`，非典型 macOS「关窗不退」习惯）——托管站点面板里的提示文案应明确
     说明这点，而不是暗示托管能在后台无限期存活。重新打开应用后，此前托管的服务不会自动恢复
     （需要重新点「发布隐藏服务」；静态托管模式下同一文件夹会复用同一身份文件，`.v0id` 地址不变）。
5. **钱包**：
   - 显示**收款地址**（+ 复制）与**余额**（与「链·挖矿」一致）。挖几个块让余额 > 0。
   - 转账：填收款地址 + 金额（+ 可选备注）→「发送」→ 提示「已提交（txid…）」；到对方/自己另一个地址，
     挖块后余额变化。空地址/非正数金额会被前端拦下并提示。
   - 私钥**永不**出现在界面/渲染层——钱包由守护进程托管，转账经令牌门控的 `/send` 由守护签名广播。

> 无 GUI 环境也能对**接线本身**有信心：所有角色/质押/钱包动作都落到本机节点 HTTP API
> （`/roles`、`/stake`、`/relay/*`、`/hs/*`、`/mine/*`、`/send`、`/balance`），其形状与鉴权已被
> 仓库工具链直接 curl 验证（POST 需 `Authorization: Bearer <userData/v0id/api.token>`；
> 激活前 `/stake`·`/unstake` 干净回 400 而非崩溃；`/hs/start` 中继不足回 409）。

## 排错

- **端口 9050 被占。** 换一个端口，两个终端**必须一致**：
  - 终端 1：`V0ID_SOCKS_PORT=9051 corepack pnpm exec tsx clients/desktop/scripts/demo-network.mjs`
  - 终端 2：`V0ID_SOCKS_EXTERNAL=9051 pnpm start`
- **页面空白 / 报 `did-fail-load`。** 依次确认：① 终端 1 还在跑（横幅在、没崩）；② 地址**粘对了**
  （就是终端 1 当次打印的那个，别用旧的）；③ 看浏览器日志：
  `~/Library/Application Support/v0id-browser/browser.log`（里面有 SOCKS 就绪、导航等记录）。
- **electron 没装 / `pnpm start` 报找不到 electron。** 在 `clients/desktop` 下重跑
  `pnpm install --ignore-workspace`。**别漏 `--ignore-workspace`**：仓库根有 `pnpm-workspace.yaml`，
  裸跑 `pnpm install` 会被「吸」进根 workspace 而**不在本目录装 electron**（装了个空）。
  （用 `npm install` 没有这个问题。）
- **窗口显示「渲染层未构建」指引页。** 没跑过 `pnpm build`（或改了 `.jsx` 没重建）。在 `clients/desktop`
  下 `pnpm build` 后重开；或用开发模式 `pnpm dev` + `V0ID_RENDERER_DEV_URL=http://localhost:5173 pnpm start`。
- **状态行不是「外部 SOCKS …」、绿点不亮。** 多半是终端 2 没设 `V0ID_SOCKS_EXTERNAL`（那样它会去自起守护并连真链，
  和本验证无关），或终端 1 的 SOCKS 端口与终端 2 给的不一致。

## 进阶：真实公网验证（可选，留作后续）

上面是离线 demo。要在**公网真链**上验证（需链上 ≥3 个中继在线）：

1. **托管你自己的隐藏服务**（任意机器，先有个本机服务，比如 `127.0.0.1:8080`）：

   ```bash
   corepack pnpm exec tsx packages/cli/src/index.ts start \
     --hs-target 127.0.0.1:8080 \
     --peers ws://<种子>:6001
   ```

   它会打印一个公网 `.v0id` 地址。

2. **用浏览器访问它**（这次**不带** `V0ID_SOCKS_EXTERNAL`，让浏览器自己起守护、加入同一张网）：

   ```bash
   cd clients/desktop
   V0ID_PEERS=ws://<种子>:6001 pnpm start
   ```

   地址栏粘贴第 1 步的 `.v0id` 地址。预期同样渲染出你那个服务的页面；不同的是这条链路跨了公网真实中继。

3. **SOCKS 出站建路应避开已知死中继**：链上目录会永久累积无法注销的历史中继（早下线/被防火墙挡的），
   过去 SOCKS 走普通 `host:port`（非 `.v0id`）出站时选路是纯随机、不看可达性，隔三差五就会随机撞上死
   中继卡满 ~6s 超时再重试，表现为「浏览器时好时不好使」。现在 `pickHops` 复用与隐藏服务同一套
   `RelayReachability` 后台探测缓存，只从已知可达集里选：起浏览器后等 ~1 分钟（后台预热探测跑一轮），
   用地址栏访问一个普通 `http://` 网址，建路应明显更快更稳，不再频繁出现长时间卡顿再失败又重试的现象。

## 无头自证（不需要 GUI，给你信心）

GUI 这一步必须你在有显示的 Mac 上亲手点。但 GUI 将走的**那条核心路径**已可无头证明（CI 友好）：

```bash
# 在 <根> 下跑（用根工具链的 tsx；clients/desktop 的 --ignore-workspace 安装里没有 tsx）：
# 完整路径 SOCKS5 → rendezvous → 隐藏服务（含正例取回 body + 负例干净失败）
corepack pnpm exec tsx clients/desktop/scripts/browser-core-test.mjs   # 期望末行 ALL PASS
```

它用真实 `curl --socks5-hostname … http://<addr>.v0id/` 取回隐藏服务 body——正是 webview 用
`proxyRules: 'socks5://…'`（Chromium 远程 DNS）的命令行等价物。
