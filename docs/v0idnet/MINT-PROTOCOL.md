# v0idChain 央行电子现金铸币厂协议（MINT-PROTOCOL）

> 配套 [INCENTIVE-PROTOCOL.md](./INCENTIVE-PROTOCOL.md) 与 [HS-PROTOCOL.md](./HS-PROTOCOL.md)。本文档讲**代币怎么从「只对内的燃料」长出「对外的通行费/交易」价值**——用户为真实服务（浏览网站、用中继、买东西）付费、服务方赚币——以及它**诚实地不防什么**、迭代去哪。所有共识改动遵循软分叉纪律（§2.4）。
>
> 实现：共识内核 = `packages/core`（`config`/`mint`/`blockchain`/`transaction`）；链下守护 = `packages/node/src/mint`（`token.ts` + `mintd.ts`）；CLI/HTTP = `packages/cli` + `packages/node/src/api.ts`。见 PR #49。

## §0 定位与威胁模型

现有代币价值**太单一、只对内**：出入口全是「燃料」（gas→矿工、挖矿产出、游戏/消息烧币）。真正让代币有**外部价值**的第③层——「用户为真实服务付费、服务方赚币」——此前几乎是断的。铸币厂就是补这一层的**统一支付层**：它同时支撑中继激励（用量回流养中继）、网站付费、（未来）市场交易。

设计由四个约束收敛而来：**央行做中间人分发** + **链下无 gas 小额** + **用量手续费回流中继** + **匿名优先**。这四者只有一个共同解——**由央行运营的盲签电子现金铸币厂（Chaumian e-cash mint）**。

**资金流**：`①用户充值(上链→托管) → ②央行发券(链下·无gas) → ③用户交券给服务方(链下·加密隧道内) → ④服务方兑现(上链·抽成回国库→养中继)`。

**为什么「过央行」比「直付」更匿名（枢纽）**：若用户直接给中继/网站打钱，链上就多一条 `用户→服务方` 的边 = 当众承认「我用了它」——直付与匿名天生矛盾。但所有人先充进**同一个**央行、再由央行付给所有服务方：`用户→央行` 不暴露用了哪个服务，`央行→服务方` 不暴露是谁付的 → 央行成了**混淆池**，匿名集 = 全体并发用户。这正是 Chaumian e-cash 的结构。

**铸币厂防：**
- **代币价值单一**（无外部效用）→ 给「用真实服务」定价、让服务方赚币。
- **中继激励不可持续**（国库有限）→ 用量抽成回流国库（§2.3），央行自筹、不烧引导池。
- **直付去匿名**（资金图谱暴露 用户↔服务方）→ 过央行混淆 +（Phase B）盲签使央行自己也无法关联。

**铸币厂明确不防（诚实声明，须在客户端显眼处保留，同 INCENTIVE §7）：**
- **铸币厂是托管方**：它给你「不可关联」，**不**给你「无需信任」。链上强制「兑现≤储备」（不能超付储备），但**超发**（发券多于储备）在联邦化/储备证明前是**残余信任**；且它可**拒绝兑现**（活性）。这与既有「中心化可信 measurer」是**同一类信任**，非新增信任类别——但风险更高：**掌 `MINT_ADDRESS` 私钥即可支配全部储备**。
- **Phase A 半匿名**：运营者能看到「谁充值、发了哪些券、谁来兑现」的映射。**密码学不可关联要到 Phase B 盲签**（§5）。
- **匿名集小 = 弱匿名**（与密码学无关）：只有一个用户在用时，混淆池混不了。
- 不防应用层去匿名、端点攻陷、全局被动对手流量关联（同 HS-PROTOCOL §0）。

## §1 设计目标与文献对齐

外部价值 → 无 gas 小额 → 用量回流可持续 → 匿名优先。与既有研究对齐（§11 署名）：
- **Chaum blind signatures / DigiCash**：盲签发行 → 央行签名却看不到序列号 → 无法关联提取与兑现。铸币厂匿名性的密码学基石（Phase B）。
- **Cashu（Bitcoin e-cash）/ BDHKE**：Blind Diffie-Hellman Key Exchange 在 secp256k1 上的极简盲签 e-cash；`@noble/curves` 已含 secp256k1 → 不引新依赖。Phase B 采用。
- **Fedimint**：联邦守护者门限签名共管铸币厂 → 去掉单运营者偿付/活性信任。终局（Phase C）。
- **Orchid / HOPR**：概率纳米支付（按流量签彩票、中继兑中奖票上链）→ 连续计量（中继带宽）的支付原语，与离散付款（网站/买东西=盲签券）互补（§4）。

## §2 链上共识（DEPOSIT / REDEEM）

链上只管一个「余额公开的金库」。共识权威在 `applyTx`，`computeState` 与 `validateChain` **共用** → 杜绝分叉。完全镜像已上线的红包/质押托管状态机。

### §2.1 两种操作
- **充值 DEPOSIT** `MINT|DEPOSIT`：转 amount 给 `MINT_ESCROW_ADDRESS`（`0x…3`）。**注意 to=托管而非自转**：旧节点把它当普通转账锁进托管（余额效果一致）→ 不静默分叉。链上储备 = 该托管地址余额 = 累计充值 − 累计兑现。备注精确匹配 `MINT|DEPOSIT`（不带关联数据，利于匿名；充值人身份取 `tx.from`）。
- **兑现 REDEEM** `REDEEM|<面额>`（amount=0，收款=`tx.to`）：**仅 `MINT_ADDRESS` 签发**（同 SLASH 仅度量者）。从托管付 `面额 − 抽成` 给服务方、`抽成` 移交国库。amount=0 是新边界，旧节点直接拒（同红包 CLAIM/REFUND、质押 UNSTAKE/SLASH）。

### §2.2 偿付不变量（链上可验证）
`redOpError` 在接纳 REDEEM 前强制 **`面额 ≤ 当前托管储备`** → 铸币厂**绝不可能超付储备**。任何人 `GET /mint/reserve`（或 `computeMintState(chain)`）即可核对「累计充值 − 累计兑现 = 当前储备 ≥ 0」= **链上储备证明**。收款不得为系统/托管地址。

### §2.3 抽成回流（用量 → 国库 → 中继）
`抽成 = floor(面额 × MINT_FEE_BPS / 10000)` 移交国库（`GENESIS_PREMINE_ADDRESS`）。国库因此被真实用量自动回填 → `reward-epoch` 发奖不再只烧有限引导池，形成**可持续闭环**（补上 INCENTIVE §4.3 的「可持续性」缺口）。守恒：`credit(托管,−面额)+credit(服务方,+净额)+credit(国库,+抽成) = 0`（净额+抽成=面额），全网供给只搬运不增发。

### §2.4 软分叉 + 激活高度（关键）
- DEPOSIT = 向托管地址的普通转账，旧节点同样锁币、余额一致 → 不静默分叉；REDEEM 是 amount=0 新边界，旧节点直接拒。
- **`MINT_ACTIVATION_HEIGHT`**：该高度前，`MINT|DEPOSIT`/`REDEEM|` 备注与 `…3` 托管地址一律按历史普通交易处理——防止升级节点重放老链时把历史交易 **retroactive** 误判成铸币操作。⚠️ 当前为**占位值 30000**，**合并/部署前必须确认它 ≥ 实时链高 + 升级窗口**（同 `STAKING_ACTIVATION_HEIGHT=16000` 的选法）。

## §3 链下守护进程（记名券 · 额度账本 · 绝不超发）

一个**中心化、诚实**的链下守护（`mintd.ts`）：单进程、单签名密钥。它不改链，只发券/验券/成形兑现交易。

### §3.1 代金券（Phase A：记名券）
券 = `{denom 面额, serial 16字节hex, sig}`。`sig` = 铸币厂 ed25519 私钥对 canonical `['v0id-token-v1', denom, serial]` 的签名。服务方用铸币厂公钥（= `MINT_ADDRESS`）**离线验签**，无需联系铸币厂即可确认「这确是央行签发的 denom 面额券」。**券是无记名持有物**：谁持有谁能花，链下流转、无 gas。

### §3.2 额度账本 + 绝不超发（核心不变量）
守护进程增量扫链（游标 = 末块 `.index`），把每笔 `MINT|DEPOSIT` 记成充值人的「可发券额度」。发券扣额度，**额度不为负 → `Σ已发 ≤ Σ充值`**。这条链下不变量**镜像**链上「兑现 ≤ 储备」——共同保证**每张券都兑得出**（偿付）。

### §3.3 兑现 + 防双花
服务方持券来兑：验签 + 防双花（`serial` 已花集合 + 批内去重）+ 收款校验 → 记 `serial` 已花 → 成形一笔 `REDEEM|<面额合计>` 交易付服务方（净额）。**先标记已花再成形**：链**不追踪 serial**，若不先标记，可对同一券成形两笔 REDEEM → 超付储备；故宁可「广播失败也已标记」（重发同 serial 被再拦），不冒超兑险。预览用 `dryRedeem`（只验券+算拆分，不消耗券）。

### §3.4 密钥纪律
`MINT_ADDRESS` 是固定常量，其**私钥绝不进仓库**（同国库/measurer；**风险更高**——掌钥即支配全部储备）。守护从 0600 钱包文件加载；只有 `.address === MINT_ADDRESS` 链才接受其 REDEEM。**部署前必须 rotate 到运营者本机生成的离线钱包**。当前占位地址生成时即弃私钥（= 激活前无人可兑现，安全默认）。

## §4 离散 vs 连续：两种付款各管一段

- **离散付款**（访问一次网站 / 买一件东西 / 打赏）→ **盲签代金券**（本协议）。一手交券、一手服务。
- **连续计量**（中继按 cell 转发带宽计费）→ **支付通道 / 概率彩票**（Orchid 模型，见 INCENTIVE §8.1）。按流量签票、中继兑中奖票。

两者**都在央行结算**（券/彩票最终兑成链上余额），共享同一个混淆池与偿付模型。故「Q2 链下无 gas」的两种实现不冲突，是各管一段。

## §5 分阶段路线

- **Phase A · 透明可信清算所**（本协议已实现）：记名券。运营者能看到 充值↔发券↔兑现 的映射 = **半匿名**，链上只见聚合流。目标：尽早把「充值→发券→花费→兑现→抽成回流」整个经济闭环跑通、验证经济模型。
- **Phase B · 盲签升级**：把 `token.ts` 的记名签名换成 **BDHKE 盲签**（Cashu 方案，secp256k1 复用 `@noble/curves`）。链上 `MINT|DEPOSIT`/`REDEEM|` 结构、托管地址、激活门控**不变**——纯链下升级。达成：运营者发券时看不见序列号 → 无法关联充值者↔兑现券 = **匿名优先**。
- **Phase C · 联邦铸币厂**（终局，先文档不实现）：单 `MINT_ADDRESS` → 多 guardian 门限签名（Fedimint 式）共管发行与储备，去掉单点偿付/活性信任。

## §6 参数表

| 常量 | 值 | 含义 |
|---|---|---|
| `MINT_ESCROW_ADDRESS` | `0x…3` | 铸币厂托管地址（金库/储备）；与红包 …1 / 质押 …2 区分 |
| `MINT_ADDRESS` | `0x4d3ea042…`（占位） | 发行/兑现授权地址；私钥离线，部署前须 rotate |
| `MINT_FEE_BPS` | 500（5%） | 兑现抽成率（基点）；抽成回国库 → 养中继 |
| `MINT_ACTIVATION_HEIGHT` | 30000（占位） | 铸币共识激活高度；部署前须确认 ≥ 实时链高 + 升级窗口 |
| `MINT_DEPOSIT_PREFIX` | `MINT\|DEPOSIT` | 充值备注（精确匹配） |
| `REDEEM_PREFIX` | `REDEEM\|` | 兑现备注前缀：`REDEEM\|<面额>` |

（均为小网络起步值、可调；改共识常量须全网一致、走软分叉。）

## §7 诚实边界（须在客户端首屏/文档显眼处保留）

1. **托管方，非无需信任**：链上强制「兑现≤储备」，但**超发**（发券>储备）与**拒兑**在联邦化前是残余信任；掌 `MINT_ADDRESS` 私钥即支配全部储备。
2. **Phase A 半匿名**：运营者可见 充值↔兑现 映射；密码学不可关联要到 Phase B 盲签。
3. **匿名集小 = 弱匿名**（与密码学无关）。
4. **部署门槛**：① rotate `MINT_ADDRESS` 到离线钱包；② 确认 `MINT_ACTIVATION_HEIGHT` ≥ 实时链高 + 升级窗口；③ 守护进程 0600 数据目录须持久 + 备份（已花 serial 丢失有被重兑风险）。

## §8 迭代路线

1. **盲签（Phase B）**：BDHKE e-cash → 真匿名。
2. **联邦（Phase C）**：多 guardian 门限共管 → 去单点信任 + 储备证明。
3. **用户面向的带鉴权发券服务**：HTTP 发券/兑现端点 + 防重放（用户签名证明拥有充值地址、券只发给合法请求者）——本轮为运营者 CLI 工具链，公开服务待本节定稿。
4. **网站付费垂直**：`hsservice.ts` 加 `--hs-price`，rendezvous 加密隧道内跑「402 需付费 → 递券 → 放行」，会合点全程只见密文。
5. **中继带宽垂直**：支付通道 / 概率彩票（Orchid），在央行结算，与现有 measurer/reward-epoch 并轨。
6. **市场接入**：既有 `market.ts`（MKT/BUY/DEL）接入铸币厂做**私密交易 + escrow + `.v0id` 交货**。

## §9 验证

- `scripts/mint-selftest.ts`（41 项）：配置向量、纯解析/兑现拆分、激活门控、DEPOSIT 开账+储备+守恒、非法充值/非授权兑现被拒、**偿付不变量**、分叉安全（`computeState ≡ validateChain ≡ replaceChain`）。
- `scripts/mint-daemon-test.ts`（27 项）：记名券签发/验签/防篡改、额度账本+**绝不超发**、REDEEM 交易成形（`parseRedeem` 回环/拆分/`verifyTransaction` 自洽）、防双花（已花/批内重复）、`dryRedeem` 预览不消耗、状态持久化恢复、偿付（储备≥未兑）。
- 运行时冒烟：`GET /mint/reserve` + `v0id mint reserve` 对真节点通过。
- ⚠️ **REDEEM 被接受本地测不了**（`MINT_ADDRESS` 私钥离线）→ 测非授权被拒 + 纯逻辑，**链上兑现接受属部署期**（同既有 SLASH 处置）。
