# 网站付费垂直 · 隧道内付费墙协议（PAYWALL）— 设计草案 DRAFT

> 状态：**草案待评审**（未实现、未接线）。这是"央行电子现金铸币厂"([MINT-PROTOCOL](./MINT-PROTOCOL.md)) 的**第一个真实消费端**：让一个 `.v0id` 隐藏服务对访问收费。评审通过后再动代码。
>
> 依赖：[HS-PROTOCOL](./HS-PROTOCOL.md)（会合隧道）+ [MINT-PROTOCOL](./MINT-PROTOCOL.md)（记名券）+ [INCENTIVE-PROTOCOL](./INCENTIVE-PROTOCOL.md)（抽成回流中继）。

---

## 0. 一句话

访客经 3 跳匿名会合隧道连上 `xxx.v0id`；服务方在**隧道内**回一个 `402 需付费`（价 + 铸币厂公钥）；访客把钱包里的**记名券**递进去；服务方**验券放行**，把连接桥接到真实网站。**会合点全程只见密文，链上无逐次访问足迹**；服务方攒券后经铸币厂 `REDEEM` 结算，抽成回流中继池。

## 1. 角色与资金流

```
用户(访客)            服务方(网站运营者)           央行(铸币厂)              链
  |  token buy(充值) ------------------------------------------------> DEPOSIT 锁进托管
  |  <----- 盲/记名券 --------------------------- issue(扣额度发券)
  |  连 xxx.v0id (会合隧道) --> [隧道内] 402 需付费
  |  ------ 递券(隧道内密文) --> 验券 -> 200 -> 桥接到 --hs-target
  |                              (攒券) ---- 提交券 ------> 验券+全局防双花
  |                                          <--- REDEEM 付 net --------> 托管付服务方,抽成回国库
```

- **用户**：`token buy` 充值换券 → 花券买服务。
- **服务方**：`--hs-price N` 托管收费站点 → 收券 → 攒够去铸币厂兑现（得 `net = 面额 − 抽成`）。
- **央行/铸币厂**：发券、兑现（唯一持 `MINT_ADDRESS` 私钥者），维护**全局已花序列号集**。
- **链**：只见聚合的 `DEPOSIT`/`REDEEM`，看不到"谁访问了哪个站点几次"。

## 2. 付费墙握手（协议核心）

握手跑在 `connectHiddenService` 建好的 **RdvChannel** 之上（端到端 AEAD，RP 解不开），发生在**把连接桥接到 `--hs-target` 之前**。是一段严格前缀的控制握手，`PAYOK` 之后信道退化为透明字节管道（既有 `bridgeChannelToSocket`）。

### 2.1 帧格式（控制阶段）

RdvChannel 收发离散 cell（净荷 ≤ ~440B）。控制消息用**长度前缀帧**：`u32be 总长 ‖ 该长度的 UTF-8 JSON`，可跨多 cell（接收方累积到够长再解析）。`PAYOK` 后不再分帧，纯透传。

### 2.2 消息（JSON，`t` 字段区分）

| 方向 | 消息 | 载荷 |
|---|---|---|
| 服务方→访客 | `PAYREQ` | `{"t":"payreq","v":1,"price":N,"mint":"0x…","unit":"$V0ID","scope":"conn","nonce":"<hex16>"}` |
| 访客→服务方 | `PAY` | `{"t":"pay","v":1,"vouchers":[[denom,"serial","sig"], …]}`（紧凑数组省字节） |
| 服务方→访客 | `PAYOK` | `{"t":"payok","v":1}` → 之后透明桥接 |
| 服务方→访客 | `PAYERR` | `{"t":"payerr","code":"insufficient|invalid|spent|expired","need":N,"got":M}` |

- `mint` = `MINT_ADDRESS`：访客据此**核验券确由本网络铸币厂签发**（`verifyToken(v, mint)`），防被引到假铸币厂。
- `nonce`：本次会话随机数（可选，绑定 PAY↔PAYREQ；真正的防重放是券的 `serial`）。
- `PAYERR{insufficient,need,got}`：访客可**再发一次 `PAY` 补券**（累加），或放弃。
- `scope:"conn"`：一次付费解锁一次会合连接（见 §5）。

### 2.3 时序

```
op: PAYREQ(price=5, mint=0x…)
cli: PAY(vouchers=[[5,serial,sig]])          # 从钱包取券,面额和 ≥ price
op:  verify(每张券) + 面额和≥price + 未花过  # 见 §3
op:  PAYOK  ──────────────────────────────►  # 之后 RdvChannel = 到 --hs-target 的透明 TCP 桥
     (若不足/非法 → PAYERR;客户端补券或断开)
```

## 3. 验券与防双花（**关键决策点**）

服务方对每张券做：① `verifyToken(v, MINT_ADDRESS)` 验签（**离线可做**，`MINT_ADDRESS` 是公开常量）；② 序列号未花过；③ 面额和 ≥ price。**难点在②的"未花过"以什么集合为准**：

- **(A) 离线乐观**（原计划默认，最低延迟）：服务方只查**自己本地的已花集** → 放行 → 攒券后再去铸币厂兑现。
  - **残余风险**：跨服务方双花——访客把**同一张券**递给服务方 A 和 B，两者各自本地都没见过 → 都放行；但最终只有先到铸币厂 `REDEEM` 的那个拿到钱，另一个**服务了却收不到款**。风险额 = 单券面额（小额付费墙可接受）。
- **(B) 在线核销**（推荐给有价值的服务）：服务方在 `PAYOK` 前调用铸币厂 `POST /mint/spend`（原子性"验签+全局标记已花+返回可兑现凭据"）→ 无双花。代价：每次访问一个**链下**铸币厂往返（非上链，快）。

**清晰的增量路径（推荐）**：
- **Phase A.1（先做，安全且简单）**：**服务方 == 铸币厂运营者**（央行自营收费站点，或首个 demo）。此时"本地已花集"**就是**全局集 → 方案 (A) 也**零双花**。直接跑通 402→券→200→桥接→兑现整链。
- **Phase A.2（再做）**：**第三方服务方**（≠铸币厂）。要么上 (B) 在线核销（无双花），要么用 (A) 承担小额残余（明写进诚实边界）。这一步顺带做"带鉴权的铸币厂发/兑券 HTTP 端点"（[MINT-PROTOCOL](./MINT-PROTOCOL.md) 待办）。
- **Phase B（盲签）与本节正交**：盲签解决"运营者无法关联充值者↔兑现券"（匿名），**不改双花模型**（双花仍靠序列号/核销）。

> **复用**：服务方的验券+本地防双花逻辑 = `MintDaemon.verifyBatch` 的子集（验签 + spent-serial 集）。第三方服务方的兑现 = 把券提交给铸币厂而非本地 `redeem`（本地 `redeem` 要 `MINT_ADDRESS` 私钥，第三方没有）。

## 4. 结算（券 → 链上 REDEEM）

- 服务方攒券到阈值 / 定时 → 向铸币厂提交券批 + 自己的收款地址。
- 铸币厂 `verifyBatch`（全局防双花）→ 成形 `REDEEM|<gross>`（`from=MINT_ADDRESS`, `to=服务方`, `amount=0`）→ 广播上链。
- 链上：托管付服务方 `net`、抽成 `fee=floor(gross×MINT_FEE_BPS/10000)` 回国库（→ reward-epoch 养中继）。**偿付强制** `Σ兑现 ≤ 储备`（[已 devnet 全环验证](../../scripts/mint-devnet-e2e.ts)）。

## 5. 计价粒度 / 找零 / 错误

- **粒度**：`scope:"conn"` = 每次会合连接付一次（简单，Phase A）。`scope:"req"`（HTTP 逐请求计费）需在隧道内解析 HTTP，**Phase A 不做**。
- **找零**：**不找零**。访客递面额和 ≥ price 即可，**多付作废**。故访客应尽量用小面额凑近整。Phase B 可加"盲签找零券"（Cashu 式），Phase A 明确不做。
- **错误**：`insufficient`→访客补券；`invalid`(验签失败)→断开；`spent`(在线模式命中已花)→拒；券**当前无过期字段**，`expired` 预留。

## 6. 匿名性质（承诺与边界）

- **链上无逐次访问足迹**：付费在加密隧道内完成，链上只有聚合 `DEPOSIT`/`REDEEM`。
- **RP 只见密文**：付费握手在端到端 AEAD 之内，会合点解不开。
- **服务方不知访客 IP / 访客不知服务方 IP**：沿用 HS 双向匿名。
- **Phase A 边界（诚实）**：记名券下**铸币厂运营者能把充值者↔兑现券关联**（半匿名）；Phase B 盲签才密码学不可关联。匿名集 = 并发用券用户。

## 6.5 延迟与速度（**设计铁律**）

**放行（PAYOK）全程链下，绝不等区块确认。** 访问一次收费站点的延迟 = **隧道 RTT + 一次 `verifyToken` 验签（µs 级）+ 查本地已花集（内存 O(1)）**，与 8 秒出块时间**无关**。链只在两处出现，且**都不在访问热路径上**：

| 何时碰链 | 在哪 | 是否卡访问 |
|---|---|---|
| ① 买券 `DEPOSIT` → 铸币厂发券 | 访问**之前**、一次性 | 否（提前备券） |
| ② 服务方 `REDEEM` 结算 | 访问**之后**、批量 | 否（离线攒券） |
| 放行验券 | 隧道内，纯密码学 | **热路径，但不碰链** |

**唯一沾链的延迟 = 买券要等 `DEPOSIT` 确认**（`mintd` 默认 6 确认 ≈ 48s）。缓解（都把它挪出访问热路径）：
- **预先买券**：浏览前一次性备好一批券，访问时钱包直接取，零等待。
- **小额降确认**：小面额券用更低确认数（浅 reorg 风险 × 小额 = 可接受）；或"待确认额度"乐观发券。
- 买券延迟与**访问延迟解耦**：访问只花已在钱包里的券，不触发任何链操作。

**热路径优化**：
- `PAYOK` 后**立即桥接**，零额外往返（`PAYREQ→PAY→PAYOK` 之后信道即透明管道）。
- `scope=conn`：一次付费解锁**整条连接**，其内后续 HTTP 请求全免费（keep-alive 友好）。
- （A.2）客户端**缓存每地址 price** → 连接时**乐观预付**（不等 `PAYREQ`），省一个隧道往返。
- 验券是纯 CPU（ed25519 verify + Set 查），单核每秒数千张，绝不是瓶颈。

**e2e 必测（对齐你的关切）**：`PAYOK` 时延、首字节时延、并**断言访问全程未出任何块**（证明放行不被打包速度拖累）。

## 8. Phase 划分

- **A.1**：`scope=conn` + 离线验券（operator==mint,零双花）+ 本地兑现 + e2e。= 最小可跑通的真实付费墙。
- **A.2**：第三方服务方（铸币厂 HTTP 发/兑券端点 + 带鉴权 + 在线核销可选）。
- **B**：盲签发/验券（BDHKE），换 `paywall` 的券格式，握手不变 → 达成匿名不可关联 + 可选盲签找零。
- **C**：联邦多签铸币厂（去单点偿付信任）。
